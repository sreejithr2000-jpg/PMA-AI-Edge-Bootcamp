1
Calendar Sync & Event Ingestion
Google Calendar and Outlook push webhook notifications on every calendar mutation (create / update / delete). Apple CalDAV is polled every 5 minutes — no webhook support. All incoming events are normalized to a canonical schema { id, title, start, end, attendees[], calendarId } and written to PostgreSQL. Duplicates are deduplicated by provider event ID with an upsert.
Calendar Sync Google / Outlook webhooks Apple CalDAV (poll) PostgreSQL
2
Conflict Detection — Sweep-Line Algorithm
On every calendar upsert, the Conflict Engine runs an O(n log n) sweep-line over the user's next 14-day window. Events are sorted by start time; an interval tree detects overlaps. Only events the user is marked "accepted" on are included. Conflicts are scored by severity (full overlap vs partial) and stored as a pending conflict record.
Conflict Engine Events DB (Postgres)
3
Layer 1 — Rule Evaluation
User-defined JSON predicate rules are loaded from PostgreSQL and evaluated first. Rules check meeting categories, organizer, attendee count, and user-set priorities (e.g. "Engineering > Cross-functional"). A rule match with confidence ≥ 0.9 proposes a resolution without calling the AI — keeping resolution latency under 80ms and avoiding Claude API costs for routine conflicts (estimated 60–70% of all conflicts).
Conflict Engine Rules DB (Postgres)
4
Layer 2 — AI Suggestion via Claude
When rules don't fire or confidence is low, the AI Orchestrator builds a context-rich prompt: conflicting events, attendee roles, user's free-time blocks (queried from Postgres), and the top-5 most similar past decisions retrieved via cosine similarity from the Vector DB. Claude is instructed to return structured JSON: { keepEvent, declineEvent, altTime, draftMessage, reasoning }. The response is validated against the schema; malformed responses fall through to Layer 3 (manual).
AI Orchestrator Claude API Vector DB (behavioral context) Free-slots (Postgres)
5
User Approval & Decline Message
The conflict suggestion is pushed to the connected client via WebSocket (sub-200ms) and simultaneously queued as a push notification fallback. The user sees the AI suggestion with the draft decline message pre-filled and editable. On approval, the Conflict Engine calls the calendar provider API to decline the event, updates the conflict record to "resolved", and fires a behavioral memory write — the decision is embedded and stored in the Vector DB for future context retrieval.
Notification Svc WebSocket (Gateway) Calendar Sync → Provider API Behavioral Memory → Vector DB
6
Attendance Tracking
A scheduled job queries all meetings that ended 5 minutes ago and publishes an attendance-check event to the Message Queue. The Notification Service consumes it and sends a push/in-app nudge. User's yes/no response is stored and fed into the Behavioral Memory Engine — attended meetings reinforce the priority model, missed ones flag patterns (e.g., recurring Thursday 3 PM misses).
Notification Svc Message Queue (scheduled job) Behavioral Memory PostgreSQL (attendance log)
7
Follow-up Capture & Async Integration Sync
After attendance confirmation, the AI Orchestrator generates meeting-context-aware follow-up suggestions (meeting title + attendees → Claude prompt). User selects/edits suggestions and chooses a destination integration. Follow-up Manager publishes sync tasks to the Message Queue — integration calls (Notion API, Jira API) are processed asynchronously to handle rate limits and partial failures without blocking the user. Integration failures are retried with exponential backoff; permanent failures surface as in-app notifications.
AI Orchestrator Follow-up Mgr Message Queue (async) Notion / Jira APIs
Tradeoff
Rules-First, AI-Second Routing
Decision: Rules evaluate before AI on every conflict.

Pro: Eliminates AI latency and cost for predictable conflicts (est. 60–70%). Sub-80ms resolution for rule matches; no Claude API call needed.

Con: Rules require user setup — cold-start users see AI-heavy, slower experience. Rules can conflict with each other and need explicit priority ordering. Maintenance burden grows as users create complex rule sets.
Tradeoff
Freemium on AI Core Features
Decision: AI conflict resolution + behavioral memory are free. Advanced integrations (Notion, Jira) and unlimited features are $9/mo.

Pro: Eliminates the paywall for the core value proposition. Users experience AI benefit before any payment friction.

Con: Claude API costs are incurred per free user per conflict. Need a hard free-tier cap (~15 AI calls/month/user) to prevent cost blowout. Estimated free-tier cost: $0.30–$0.60/user/month at current Claude pricing.
Tradeoff
PWA-First vs Native App
Decision: PWA for MVP, native shell added later.

Pro: Single codebase, instant updates, no app store friction or review delays.

Con: iOS Safari has limited push notification support for PWAs (added in iOS 16.4, but unreliable). Background calendar sync requires server-side polling for the Apple CalDAV integration. These gaps create a weaker experience for iPhone users — the most valuable mobile segment for this product.
Insight
Webhook vs Polling — Protocol Asymmetry
Google Calendar and Outlook support push webhooks (events arrive in <2s). Apple CalDAV has no webhook support — polling every 5 minutes is the only option. This creates a two-tier experience: Google/Outlook users get near-instant conflict detection, Apple users get up to 5-minute lag.

Implication: Marketing and onboarding should set expectation for Apple users. Mitigated by polling more aggressively (1 min) for users with active conflicts.
Insight
Behavioral Memory via Vector Embeddings
User decisions are encoded as vector embeddings rather than stored as raw logs. At AI prompt time, a cosine similarity search retrieves the 5 most contextually relevant past decisions.

This keeps prompt length bounded regardless of how long the user has used the product, avoids storing PII in prompts, and enables personalization that improves over time without any explicit model fine-tuning. Cold-start is handled by an onboarding preferences survey that seeds initial embedding clusters.
Insight
Structured JSON Output from Claude
Claude is constrained to return a typed schema: { keepEvent, declineEvent, altTime, draftMessage, reasoning }. This makes the conflict resolver fully deterministic — no text parsing, no ambiguity.

A validation layer catches malformed or out-of-schema responses and falls back to Layer 3 (manual resolution) rather than presenting invalid data to the user. Claude's constrained output mode also measurably reduces hallucination in structured fields like altTime.
High Risk External Dependency
Calendar API Rate Limits
Google Calendar imposes 1M queries/day per project and 10 req/sec per user. Webhook bursts during peak business hours (9–10 AM) could exhaust per-user quotas. Mitigation: Per-user request budgets tracked in Redis. Exponential backoff with jitter on all calendar API calls. Batch reads during off-peak resync windows. Separate GCP projects per 500 users to distribute quota.
High Risk Privacy / Compliance
Calendar PII in AI Prompts
Meeting titles, attendee names, and descriptions sent to Claude may contain sensitive PII or confidential project names. Mitigation: Pseudonymize attendee names before sending (replace with role hashes). Allow users to flag calendar categories as "private" — excluded from AI prompts. Anthropic zero-retention API mode for all prompts. DPA (Data Processing Agreement) executed with Anthropic before launch.
High Risk Core Feature Reliability
AI Hallucinating Invalid Time Slots
Claude could suggest a reschedule time that conflicts with another event or falls outside working hours. Mitigation: Hard architectural constraint — the AI only receives a pre-validated list of free blocks as selectable options. Free-slot computation is done by the service layer, not the AI. The AI picks from the list; it cannot invent new times. This eliminates the hallucination vector entirely for this field.
High Risk UX / Trust
Unsolicited Decline Messages Sent
A bug or misunderstood UX could cause a decline message to send before user review, damaging professional relationships. Mitigation: Hard constraint in MVP: decline messages are always shown in a review modal before sending. Zero autonomous sends in MVP phase. A separate "allow autonomous sends" toggle in settings defaults to OFF. The send action requires an explicit secondary confirmation click. No "undo" period — the constraint is prevention, not recovery.
Medium Risk Compliance
FERPA Exposure for Student Users
Students using .edu Google/Outlook accounts have FERPA protections that may extend to meeting data (course discussions, advisor meetings, grade reviews). Mitigation: Separate data processing policies for .edu email domains. Opt-in consent flow for students that explicitly discloses AI processing. No training on student data under any circumstances. FERPA BAA required before pursuing institutional (B2B) deals with universities.
Medium Risk Cost
Claude API Cost at Free-Tier Scale
Each conflict resolution: ~$0.01–0.04 depending on context length. At 10K free users averaging 5 AI resolutions/month = $500–$2,000/month in AI costs before meaningful revenue. Mitigation: Rules-first routing reduces AI calls by ~65%. Free tier hard cap: 15 AI calls/month (soft UI warning at 10). Prompt caching for behavioral memory context (significant savings on repeated context blocks). Monitor cost-per-user weekly; adjust cap if CAC math breaks.
Low Risk Infrastructure
WebSocket Scaling Under Load
Real-time conflict push via WebSocket requires sticky sessions or a shared pub/sub layer for horizontal scaling. A single instance works for MVP but won't survive 10K+ concurrent connections. Mitigation: Redis Pub/Sub as the WebSocket message bus from day one — all instances subscribe to user-scoped channels. This makes horizontal scaling stateless. Connection limits per instance (500) with load balancer health checks. Graceful degradation: fall back to polling if WebSocket drops.
Low Risk UX / Onboarding
Behavioral Memory Cold Start
New users have zero behavioral embeddings — the AI operates on generic rules for the first 2–4 weeks, which reduces personalization quality and increases the chance of a wrong suggestion that erodes trust early. Mitigation: Onboarding survey (5 questions: work hours, meeting priority, common categories, decline comfort level) generates seed embeddings that approximate a 2-week usage profile. Transparent UI: show users "AI is still learning your preferences" during the early period with a progress indicator that reflects actual decision count.
